# Red-Team Security Adversary Plan Review

## Findings

### 1. Configured/PATH binary discovery is an unguarded code-execution boundary

- **Severity:** High
- **Exact phase/section:** Phase 1 — Architecture / Functions and Interfaces Checklist; Phase 2 — Requirements / Implementation Steps
- **Flaw:** The plan executes a user-supplied binary during schema generation and later discovers the runtime binary from configuration or `PATH`, but the only identity check is a self-reported `0.144.5` version. It defines neither canonical-path pinning, executable ownership/permission checks, symlink-swap protection, user confirmation when the resolved executable changes, nor a least-privilege environment inheritance policy.
- **Concrete failure:** An attacker places a fake `codex` earlier in the IDE process `PATH`, replaces a configured symlink after validation, or points the setting at a trojan that prints `0.144.5`. The plugin executes it as the IDE user during generation or startup; the child can read the project and inherited environment, exfiltrate credentials, and return protocol traffic that looks valid.
- **Evidence:** Phase 1 explicitly invokes a “user-supplied 0.144.5 binary” and only requires rejection of non-0.144.5 output (`plans/260716-2242-codex-intellij-functional-parity/phase-01-start.md:27`, `plans/260716-2242-codex-intellij-functional-parity/phase-01-start.md:53`). Phase 2 permits configured/`PATH` discovery and immediately launches the result (`plans/260716-2242-codex-intellij-functional-parity/phase-02-app-server-lifecycle-and-capabilities.md:21`). Its process step promises not to leak the environment but never defines which environment is inherited (`plans/260716-2242-codex-intellij-functional-parity/phase-02-app-server-lifecycle-and-capabilities.md:59`, `plans/260716-2242-codex-intellij-functional-parity/phase-02-app-server-lifecycle-and-capabilities.md:62`).
- **Suggested fix:** Add a Phase 1/2 `CodexBinaryTrustPolicy`: resolve once to a canonical regular executable; reject symlinks or revalidate inode/device/owner immediately before launch; pin and display executable SHA-256 plus path; require confirmation on path/hash change; disable silent `PATH` fallback after first selection; launch with an explicitly documented environment allowlist/minimal denylist and never expose inherited values to diagnostics. Add fake-binary tests for PATH precedence, symlink replacement, version spoofing, permission/owner changes, and environment capture.

### 2. Schema hashes attest to attacker-produced output, not artifact provenance

- **Severity:** High
- **Exact phase/section:** Phase 1 — Architecture / Implementation Steps / Risk Assessment
- **Flaw:** The same untrusted executable generates stable and experimental schemas, after which the plan hashes and commits those outputs. A SHA-256 manifest proves only that the generated files did not change after generation; it does not prove that the generator was an authentic Codex 0.144.5 artifact or that the schemas match a trusted release. Recording the tool arguments adds no authenticity.
- **Concrete failure:** A compromised or counterfeit binary emits a poisoned schema that classifies an attacker-controlled method or field as stable, and prints the expected version. Review and CI reproduce the attacker-authored hashes, so `CapabilityRegistry` later treats the poisoned contract as approved and sends privileged requests to the same compromised child.
- **Evidence:** The plan says generated bundles and their SHA-256 manifest are created by the user-supplied binary (`plans/260716-2242-codex-intellij-functional-parity/phase-01-start.md:21`, `plans/260716-2242-codex-intellij-functional-parity/phase-01-start.md:27`) and records hashes/tool arguments from that run (`plans/260716-2242-codex-intellij-functional-parity/phase-01-start.md:59`). The research itself states that the cited evidence snapshots live under temporary paths and that the app-server is experimental/version-specific (`plans/reports/260716-2240-app-server-capability-research.md:15`); the test research confirms that no generated schema currently exists in the repository (`plans/reports/260716-2240-parity-test-research.md:19`).
- **Suggested fix:** Make provenance a G0 release blocker: record the canonical generator path and binary digest, verify it against an independently trusted release checksum/signature or an organization-controlled artifact manifest, and compare generated schemas against a reviewed committed baseline. Require an explicit two-part update diff—binary provenance change and schema semantic delta—with stable-to-X reclassification checks. Do not let the generator author its own trust root.

### 3. Approval identity omits the process epoch, enabling stale-click approval of a new request

- **Severity:** High
- **Exact phase/section:** Phase 5 — Architecture / Functions and Interfaces Checklist; Phase 8 — Architecture / Functions and Interfaces Checklist
- **Flaw:** `ApprovalCoordinator` is keyed by server-request ID plus project/thread/turn/item, but not by app-server process epoch or a request-payload fingerprint. Phase 8 later introduces process epochs, yet its public approval and gateway interfaces still use a bare `ServerRequestId`. Server-generated JSON-RPC request IDs can repeat after restart.
- **Concrete failure:** Process A displays approval ID `7` and then crashes. Process B starts and emits a different destructive approval with ID `7` for the same resumed thread/item identifiers. A delayed click or queued callback from A calls `respond(7, approve)` on B, approving work the user never inspected.
- **Evidence:** The Phase 5 key is explicitly “server-request + project/thread/turn/item” (`plans/260716-2242-codex-intellij-functional-parity/phase-05-execution-approvals-review-and-diff.md:27`), and its decision interface accepts only `id` (`plans/260716-2242-codex-intellij-functional-parity/phase-05-execution-approvals-review-and-diff.md:50`). The gateway likewise responds by bare `ServerRequestId` (`plans/260716-2242-codex-intellij-functional-parity/phase-02-app-server-lifecycle-and-capabilities.md:50`), while epoch handling is deferred to Phase 8 (`plans/260716-2242-codex-intellij-functional-parity/phase-08-resilience-parity-audit-and-documentation.md:28`, `plans/260716-2242-codex-intellij-functional-parity/phase-08-resilience-parity-audit-and-documentation.md:54`).
- **Suggested fix:** Introduce `ServerRequestKey(epoch, id)` in Phase 2 and carry it through the adapter, coordinator, banner action, and gateway response. Bind approval UI to an immutable normalized payload hash and origin tuple; before sending, atomically verify current epoch, unresolved state, and identical fingerprint. On disconnect, invalidate the epoch before publishing reconnect state. Add a fixture where the same server request ID and thread IDs recur after restart and assert the old action cannot emit any response.

### 4. Generic server-config writes can alter execution-sensitive policy without an allowlist

- **Severity:** High
- **Exact phase/section:** Phase 6 — Requirements / Functions and Interfaces Checklist / Implementation Steps
- **Flaw:** `ServerConfigController.write(path,value,expectedVersion)` is a generic write primitive. The plan handles optimistic concurrency and managed overrides, but it never constrains writable key paths, value types, provenance, or security-sensitive settings; only the separate `CodexSettingsState` has a persistence whitelist. The capability research establishes that configuration includes managed requirements and permission-related surfaces, so version matching is not authorization.
- **Concrete failure:** A dispatcher bug, crafted command invocation, or future UI control passes an unintended writable path/value. The plugin successfully changes an execution-affecting Codex setting instead of merely changing the intended display/model preference, and the next turn runs with weaker policy or an attacker-controlled integration configuration.
- **Evidence:** Phase 6 combines server configuration and settings, specifying layered reads and `expectedVersion` writes (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:23`), then exposes the unrestricted `write(path,value,expectedVersion)` interface (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:56`). Its tests cover conflicts and managed overrides, not forbidden paths or privilege-changing values (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:81`). The app-server research groups permissions, config, and managed requirements in the same stable surface (`plans/reports/260716-2240-app-server-capability-research.md:36`).
- **Suggested fix:** Replace the generic UI-facing method with typed commands over an explicit writable-key allowlist and schema-validated value types. Classify security-sensitive paths as read-only or require a dedicated confirmation showing old/new values and effect; never derive a config path from server/model/MCP text. Enforce managed-policy checks before constructing the RPC, not only after a conflict response. Add negative tests for unknown/nested paths, path-encoding variants, type confusion, policy weakening, and stale-version retries.

### 5. Direct MCP tool calls and elicitation lack a per-invocation trust/consent contract

- **Severity:** High
- **Exact phase/section:** Phase 6 — Requirements / Functions and Interfaces Checklist / Implementation Steps
- **Flaw:** The plan enables direct MCP tool/resource calls and one-shot elicitation but specifies only failure isolation and secret-free rendering. It does not require an immutable preview of server identity, tool name, argument payload, requested data, or side effects; it also does not state that rendered MCP content can never trigger a call. “Typed” is not an authorization boundary.
- **Concrete failure:** A malicious or compromised MCP server publishes deceptive tool metadata or an elicitation asking for sensitive project/user data. The user clicks what appears to be a benign action, or an event route invokes the direct-call handler with attacker-controlled arguments, causing data disclosure or a side-effecting tool execution outside the approval UI.
- **Evidence:** MCP scope explicitly includes “resource read, direct tool/progress, typed elicitation” (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:24`) and the controller checklist requires resource/tool plus one-shot elicitation (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:57`). The implementation step tests OAuth/elicitation/failure but names no argument-preview or authorization invariant (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:65`). Research classifies direct tool calls and bidirectional elicitation as available app-server methods, not automatically safe operations (`plans/reports/260716-2240-app-server-capability-research.md:35`).
- **Suggested fix:** Route every user-initiated direct tool call and elicitation response through a one-shot consent object keyed by process epoch, MCP server ID, tool/resource ID, and immutable argument/request hash. Show server identity, destination/resource URI, bounded redacted arguments, and whether the tool is read-only or side-effecting; deny on unknown metadata, mutation after preview, timeout, disconnect, or server removal. Prohibit calls originating from rendered links/text. Add deceptive-name, argument-swap, stale-flow, oversized payload, and secret-field fixtures.

### 6. Worktree/project target canonicalization does not impose an allowed-root boundary

- **Severity:** Medium
- **Exact phase/section:** Phase 7 — Requirements / Functions and Interfaces Checklist / Implementation Steps
- **Flaw:** The plan canonicalizes target paths and asks for confirmation, but canonicalization only normalizes a path; it does not prove that the directory belongs to the current project or an approved worktree base. It also never mandates argv-safe Git invocation, `--` separation for option-like names, post-creation symlink revalidation, or bounded cleanup ownership.
- **Concrete failure:** A crafted/accidentally selected canonical path outside the project becomes a new thread cwd, allowing the agent to read or modify unrelated home/system files. If worktree branch/path data is passed ambiguously to `git`, option-like values can alter Git behavior; a symlink swap between validation and thread start can redirect the cwd outside the approved tree.
- **Evidence:** Phase 7 defines `/project` as a root picker and `/worktree` as a client-owned filesystem workflow (`plans/260716-2242-codex-intellij-functional-parity/phase-07-agents-and-advanced-execution-targets.md:23`). `ExecutionTargetRegistry` returns `Project(root)` or `Worktree(path)` without an ownership/root invariant (`plans/260716-2242-codex-intellij-functional-parity/phase-07-agents-and-advanced-execution-targets.md:55`), and the implementation step lists canonical path, confirmation, collision, and dirty handling but no containment or process-invocation rules (`plans/260716-2242-codex-intellij-functional-parity/phase-07-agents-and-advanced-execution-targets.md:63`). The test research says worktree support should be enabled only after schema/runtime proof or remain unavailable (`plans/reports/260716-2240-parity-test-research.md:86`).
- **Suggested fix:** Define allowed roots separately from canonicalization: current IntelliJ content roots for `/project`, and a configured worktree parent owned by this plugin/workflow for `/worktree`; external targets require a high-friction explicit override that shows the full canonical path and access implications. Revalidate file key/canonical containment immediately before thread start. Invoke Git through structured argv with fixed subcommands, `--`, strict branch/ref validation, no shell, sanitized environment, and a creation receipt that alone authorizes cleanup. Add `..`, symlink-race, option-prefix, nested-repo, external-path, and cleanup-boundary tests.

### 7. Redaction is implemented after diagnostics and feedback become runnable

- **Severity:** High
- **Exact phase/section:** Phase 2 — Implementation Steps; Phase 6 — Implementation Steps; Phase 8 — Related Code Files / Implementation Steps
- **Flaw:** Phase 2 creates redacted diagnostics and Phase 6 enables feedback upload, but the only named structured exporter/redaction test corpus appears in Phase 8. No shared redaction boundary or adversarial secret-corpus test is owned by the earlier phases. Because every phase must leave `runIde` useful, Phase 6 can transmit diagnostics before the planned privacy implementation is verified.
- **Concrete failure:** App-server stderr or protocol metadata contains a bearer token, home path, prompt/context fragment, MCP payload, or memory data. Phase 2 stores it in ad hoc diagnostics; Phase 6’s feedback preview misses the format and uploads it with consent that was based on an incomplete preview. Phase 8’s later exporter tests cannot undo the disclosure.
- **Evidence:** Phase 2 introduces “bounded redacted diagnostics” without a redaction component or secret-corpus test (`plans/260716-2242-codex-intellij-functional-parity/phase-02-app-server-lifecycle-and-capabilities.md:59`). Phase 6 requires feedback preview/redaction/consent and implements feedback before the exporter exists (`plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:54`, `plans/260716-2242-codex-intellij-functional-parity/phase-06-settings-mcp-and-slash-commands.md:63`). `DiagnosticsExporter` and its secret-corpus test are first created in Phase 8 (`plans/260716-2242-codex-intellij-functional-parity/phase-08-resilience-parity-audit-and-documentation.md:36`, `plans/260716-2242-codex-intellij-functional-parity/phase-08-resilience-parity-audit-and-documentation.md:42`, `plans/260716-2242-codex-intellij-functional-parity/phase-08-resilience-parity-audit-and-documentation.md:65`).
- **Suggested fix:** Move a shared `StructuredDiagnosticEvent` + `RedactionPolicy` and adversarial corpus into Phase 2; make raw stdout/stderr/protocol/model/MCP bodies non-exportable by type. Phase 6 feedback must consume only a `RedactedBundle` produced by that policy, compare the exact preview bytes/hash with the uploaded bytes, and require separate consent per destination. Phase 8 may add destination/export UI and broader stress coverage, but not the first enforceable redaction layer.

## Status

**REJECT — security plan revision required before implementation.**

## Summary

7 findings: 6 High, 1 Medium, 0 Critical. Release-blocking themes are executable/schema provenance, epoch-safe approvals, generic security-sensitive configuration writes, MCP invocation consent, filesystem target containment, and moving redaction ahead of feedback.

## Concerns

- No root `README.md` exists yet; the platform research independently records the same missing repository context (`plans/reports/260716-2240-intellij-platform-research.md:150`).
- The plan needs an explicit trust-boundary table for IDE user/UI, plugin, Codex executable, app-server messages, MCP servers, Git/worktree filesystem, and exported diagnostics.
- The plan needs named negative acceptance gates for binary spoofing, epoch/request-ID reuse, config-policy weakening, MCP argument substitution, external target containment, and redaction bypass.
